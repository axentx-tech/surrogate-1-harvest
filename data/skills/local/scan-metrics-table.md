## Overview
The pattern introduces a `scan_metrics` table in the PostgreSQL database to store start/end timestamps, duration, status, and error messages for each security scan. This centralizes metric data, improves observability, and supports compliance reporting.

## Schema Definition
```sql
CREATE TABLE IF NOT EXISTS scan_metrics (
    id SERIAL PRIMARY KEY,
    scan_id UUID NOT NULL REFERENCES scans(id) ON DELETE CASCADE,
    started_at TIMESTAMPTZ NOT NULL,
    ended_at TIMESTAMPTZ NOT NULL,
    duration_seconds DOUBLE PRECISION GENERATED ALWAYS AS (EXTRACT(EPOCH FROM (ended_at - started_at))) STORED,
    status VARCHAR(20) NOT NULL CHECK (status IN ('pending','running','completed','failed')),
    error_message TEXT,
    created_at TIMESTAMPTZ DEFAULT NOW()
);
```

## Implementation Steps
1. **Alembic Migration**: Add a new migration script (e.g., `2026xxxx_add_scan_metrics_table.py`) with the above `CREATE TABLE` statement.
2. **Backend Instrumentation**: Update FastAPI scan endpoints to insert a row at scan start (`status='running'`, `started_at=now()`), and update it on completion with `ended_at`, final `status`, and any `error_message`.
3. **Metric Export**: Expose Prometheus metrics (`scan_duration_seconds`, `scan_status_total`) derived from the `scan_metrics` table.
4. **Dashboard**: Add Grafana panels visualizing average scan duration over time and failure rates.
